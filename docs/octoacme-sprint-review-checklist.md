# OctoAcme — Sprint Review & Demo Checklist

## Purpose
Ensure every sprint review / demo is well-prepared, consistently run, and produces actionable follow-up items.

## Before the Review (1–2 days prior)

### Preparation
- [ ] Confirm all items marked "Done" meet the Definition of Done
- [ ] Identify which items will be demoed and in what order
- [ ] Assign a demo presenter for each item (Developer or QA)
- [ ] Prepare demo environment (staging or local with production-like data)
- [ ] Draft the agenda and share it with attendees
- [ ] Confirm meeting invite includes correct stakeholders and Product Manager

### QA Sign-off
- [ ] All acceptance criteria verified for demo items
- [ ] No outstanding critical or high-severity defects blocking the demo
- [ ] Smoke tests passed on the demo environment

---

## During the Review (~45–60 minutes)

### Opening (5 min)
- [ ] PM or Project Manager opens with sprint goal and context
- [ ] Confirm any items NOT completed and briefly explain why

### Demo Walkthrough (25–35 min)
- [ ] Walk through each completed item against its acceptance criteria
- [ ] Show real functionality — avoid slides as a substitute for working software
- [ ] Invite questions and feedback after each item

### Metrics & Progress (5–10 min)
- [ ] Share velocity or burndown for the sprint
- [ ] Review key success metrics relevant to the sprint goal
- [ ] Flag any risks or blockers surfaced during execution

### Stakeholder Feedback (5–10 min)
- [ ] Collect feedback and capture it as notes or GitHub issues
- [ ] Identify any scope change requests and route them to backlog

---

## After the Review

### Follow-up Actions
- [ ] Log feedback items as backlog issues with appropriate labels
- [ ] Update the Risk Register if new risks were raised
- [ ] Share a brief sprint summary with stakeholders (use Weekly Status Template)
- [ ] Confirm items accepted by Product Manager as "Done"
- [ ] Carry over incomplete items to the next sprint with updated estimates

### Retrospective Trigger
- [ ] If significant issues surfaced during the review, add them as retrospective input items

---

## Tips
- Keep demos focused on the user value delivered, not implementation details.
- Limit the meeting to the agreed timebox; use async follow-up for detailed discussions.
- Record the session if remote stakeholders cannot attend.
